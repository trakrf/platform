The test rig

A motorized turntable rotates a vertical stack of four RFID tags past a two-antenna reader portal. Every pass that the engine judges as an "entry" fires the alarm beacon.

End-to-end actuation — caught on camera

The two states below prove the complete loop physically fires and releases — not just that database records were written.

tag passes portal above signal threshold → geofence engine fires (7,315 events recorded) → alarm dispatched to wireless relay over MQTT → red beacon ON → tag rotates away · signal drops for >5 s · times out → relay releases → beacon OFF

Per-tag results

All four tags ride the same turntable but sit at different heights, so each couples to the antennas differently. A tag only counts as "present" while its signal is stronger than the −65 dBm gate; it re-fires each time it drops out and returns. The result is a clean, predictable fan-out of re-trigger rates.

The spread is expected, not a fault. The strongest-coupled tag (top of the stack) clears the detection threshold on nearly every rotation and re-fires often; the weakest only clears it occasionally and re-fires rarely. With the threshold fixed, signal strength alone cleanly orders the four — confirming both the sensitivity gate and the auto-reset timer are doing their jobs.

Tuning knob for future runs. The detection threshold currently slices through the middle of the tag stack, which is what produces the nice spread. Lowering it would make every tag fire on every rotation; raising it would let only the top tag fire. The behavior is a configurable setting, not a hardware limitation.

Stability over the full run

Across 32 consecutive hourly windows the combined fire rate held flat between 220 and 266 per hour, with all four tags present in every single window — no dead hours, no gaps, no bursts. Of 7,314 consecutive alarm intervals, only 9 (0.12%) occurred under two seconds apart, confirming the re-trigger latch is suppressing duplicate alarms as intended rather than re-firing on every read.
